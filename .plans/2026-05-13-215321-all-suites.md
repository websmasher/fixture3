# Goal

Add explicit all-suite execution for projects that split behavior checks across many suites.

The desired end state is:

- `fixture3 check --suite <name>` keeps running one named suite.
- `fixture3 check --all` runs every suite in `fixture3.yaml`.
- `fixture3 status --suite <name>` reports one named suite.
- `fixture3 status --all` reports every suite in `fixture3.yaml`.
- `fixture3 status` without either flag remains accepted for backward compatibility and reports every suite.
- `--suite` and `--all` are mutually exclusive wherever both exist.
- Aggregate `check --all` exit code is `2` if any suite errors, `1` if no suite errors and at least one suite differs, and `0` only when every suite matches.

# Approach

1. Update CLI argument parsing in `crates/fixture3/src/args.rs`.
   - Add `--all` to `check`.
   - Add `--all` to `status`.
   - Enforce `--suite` conflicts with `--all`.
   - Keep `check` requiring exactly one target: `--suite` or `--all`.
   - Keep `status` allowing no target as a compatibility alias for all suites.

2. Add a target resolver in the app layer.
   - Create a small internal target type for resolved suite selection.
   - Resolve `check --suite <name>` to one suite.
   - Resolve `check --all` to every suite in manifest order.
   - Resolve `status --suite <name>` to one suite.
   - Resolve `status --all` and bare `status` to every suite in manifest order.

3. Reuse the existing single-suite behavior.
   - Keep `run_check` as the implementation for one suite.
   - Add an aggregation wrapper that calls `run_check` for each selected suite.
   - Do not duplicate command execution, normalization, diff, metadata, or storage logic.

4. Define aggregate output.
   - Print one block per suite.
   - Prefix each block with `suite: <name>`.
   - Preserve each suite's received path, diff path, fixture count, and status.
   - For suite errors, include `suite: <name>` and the error text in stderr.

5. Define aggregate exit-code precedence.
   - If any selected suite returns a tool/runtime error, return `2`.
   - Else if any selected suite differs, return `1`.
   - Else return `0`.

6. Extend self fixtures.
   - Add a multi-suite all-match case.
   - Add a multi-suite mixed match/different case.
   - Add a multi-suite error case.
   - Add status `--all` coverage.
   - Add CLI invalid-target coverage for `check --suite <name> --all`.

7. Extend manifest verification.
   - Add `--all` to the `check` and `status` CLI command contracts.
   - Add help text requirements for aggregate exit-code behavior.
   - Add self-harness expected outcomes for the new fixture cases.
   - Ensure `scripts/verify-all.sh` remains the only completion signal.

# Key Decisions

- `diff --all` is not part of this plan.
  - Reason: `check --all` is the useful automation path, and `diff --refresh --all` can be added later with the same target resolver if needed.

- `status` without flags remains an all-suite alias.
  - Reason: current behavior already lists every suite, and removing it would be a needless breaking change.

- Suite iteration uses manifest order.
  - Reason: manifest suites are currently parsed into a `BTreeMap`, so output is deterministic by suite name.

- Aggregate output stays line-oriented.
  - Reason: current output is line-oriented and fixture-friendly; changing to a new format would create unrelated churn.
